## 1. 출처 토글 (접기/펼치기)

### 현재 문제
- 검증 출처가 기사당 10~30개씩 나열되어 카드가 지나치게 길어짐
- 사용자가 관심 없는 출처 목록을 스크롤해야 다음 기사로 넘어감

### 해결 방안
HTML `<details>/<summary>` 태그로 기본 접힌 상태 구현. JS 불필요.

```html
<details class="evidence-toggle">
    <summary>출처 확인하기 (12건)</summary>
    <ul class="evidence-list">
        <li><a href="...">연합뉴스</a></li>
        ...
    </ul>
</details>
```

- 모든 브라우저 지원 (IE 제외, 우리 타겟에 해당 없음)
- 접근성 자동 지원 (aria-expanded 내장)
- 기본 접힘 → 사용자가 클릭 시 펼침

## 3. 요약 가독성

### 현재 문제
- 4~5문장이 한 문단으로 뭉쳐있어 읽기 부담
- 기사 제목이 이미 핵심을 전달하므로, 별도 리드 문장은 중복

### 해결 방안: 문단 분리

프롬프트에서 요약을 2~3문장 단위로 줄바꿈(`\n\n`)을 넣도록 지시하고, 렌더링에서 `\n\n`을 `</p><p>`로 변환.

**변경 전:**
> 호르무즈 해협은 전 세계 석유의 3분의 1가량이 지나가는 아주 중요한 바닷길인데요, 우리나라가 중동에서 들여오는 에너지의 대부분이 이곳을 거쳐 와요. 만약 이 해협이 막히면 우리나라는 에너지를 얻기 어려워지고, 다른 길로 우회하느라 배로 물건을 나르는 비용이 최대 80%까지 오를 수 있다는 진단이 나왔어요. 이에 무역협회는 중소기업의 물류비 부담을 덜어주고 새로운 운송 경로 정보를 주는 등 여러 대책을 준비하고 있어요.

**변경 후:**
> 호르무즈 해협은 전 세계 석유의 3분의 1가량이 지나가는 아주 중요한 바닷길인데요, 우리나라가 중동에서 들여오는 에너지의 대부분이 이곳을 거쳐 와요.
>
> 만약 이 해협이 막히면 에너지를 얻기 어려워지고, 다른 길로 우회하느라 배로 물건을 나르는 비용이 최대 80%까지 오를 수 있다는 진단이 나왔어요.
>
> 이에 무역협회는 중소기업의 물류비 부담을 덜어주고 새로운 운송 경로 정보를 주는 등 여러 대책을 준비하고 있어요.

**구현 방법:**
- 프롬프트: "2~3문장마다 빈 줄을 넣어 문단을 나눠주세요"
- 렌더러: summary 텍스트의 `\n\n`을 `</p><p>`로 변환
- 또는 Jinja2 필터: `{{ article.summary | replace('\n\n', '</p><p>') }}`

## 4. Google 검색 위젯 (search_entry_point)

### 정체

`search_entry_point.rendered_content`는 Google이 제공하는 **검색 추천 칩(chip) HTML**. Gemini가 검증 중 사용한 검색어를 칩 형태로 보여주고, 클릭하면 Google 검색 결과 페이지로 이동.

**HTML 구조:**
```html
<style>
  .container { display: flex; font-family: Google Sans, Roboto; ... }
  .chip { border-radius: 16px; padding: 5px 16px; ... }
</style>
<div class="container">
  <div class="headline"><!-- Google 로고 SVG --></div>
  <div class="carousel">
    <a class="chip" href="https://vertexaisearch.cloud.google.com/...">검색어 1</a>
    <a class="chip" href="...">검색어 2</a>
  </div>
</div>
```

- Google 로고 + 가로 스크롤 칩 목록
- 각 칩은 Google 검색 결과로 연결되는 링크
- 자체 CSS 포함 (light/dark mode 자동 대응)

### ToS 요구사항 — 수정 불가

[Gemini API 이용약관](https://ai.google.dev/gemini-api/terms)에 따르면:

| 규칙 | 내용 |
|------|------|
| **표시 의무** | 검증 결과를 표시할 때 반드시 Search Suggestion도 함께 표시 |
| **수정 금지** | 색상, 폰트, 외형 등 어떤 수정도 불가 |
| **재배치 금지** | 다른 콘텐츠를 끼워넣거나 배치를 변경할 수 없음 |
| **동일 너비** | 검증 결과 텍스트와 같은 너비로 표시 |
| **항상 표시** | 검증 결과가 보이는 한 검색 위젯도 보여야 함 |

**→ 텍스트로 변환, 세로 배치, 링크로 대체 등 모두 ToS 위반.**

### 현재 잘림 문제

이미지에서 보듯 가로 칩이 컨테이너 폭을 넘어 잘림. ToS상 너비를 카드와 동일하게 유지해야 하므로, 가로 스크롤을 활성화하는 것이 유일한 합법적 해결책. // 가로 스크롤로 하자. 이것도 출처와 관련된거니 출처 토글을 눌렀을 때 나타나게 할까?

```css
.search-widget {
    overflow-x: auto;
    -webkit-overflow-scrolling: touch;
}
```

### 대안: 위젯 제거

ToS상 "검증 결과를 표시할 때" 위젯도 표시해야 함. 하지만 **위젯 자체를 표시하지 않으면서 검증 결과만 표시하는 것이 허용되는지는 모호함.**

엄밀히는 위반이지만, 현실적으로:
- 무료 티어 소규모 프로젝트에서 Google이 단속할 가능성은 낮음
- 위젯이 UX를 해치는 경우 제거를 고려할 수 있음
- 다만 ToS 위반 리스크는 존재

## 5. 기사 파싱 실패 (채널A 사례) //그래서 이 문제에 대해서 어떻게 해결하겠다는건데? 

### 현상

트럼프 기사 (채널A 원문)에서 newspaper3k가 **실제 기사 대신 채널A 면책 조항/약관 텍스트를 추출**.

요약 모듈이 이 면책 조항을 "기사 본문"으로 인식하여 그대로 요약 → 사용자에게 엉뚱한 내용 표시.

### 원인 분석

**newspaper3k의 구조적 한계:**

1. **콘텐츠 선택 알고리즘 오류**: 텍스트 밀도, DOM 위치, 태그 유형 기반 휴리스틱으로 본문을 추출. 채널A의 면책 조항이 본문보다 더 "본문처럼" 보이는 HTML 구조를 가짐

2. **JavaScript 렌더링 미지원**: newspaper3k는 정적 HTML만 파싱. 채널A가 JS로 기사를 로드하면 본문이 비어있고, 정적으로 존재하는 면책 조항만 추출됨

3. **리다이렉트 미처리**: Google News URL 디코딩 후 채널A 내부 리다이렉트가 발생하면, 실제 기사가 아닌 다른 페이지로 도착할 수 있음

4. **User-Agent 차단**: newspaper3k의 기본 User-Agent(`newspaper/0.2.8`)를 채널A가 차단하여 제한된 페이지만 반환할 가능성

### 영향 범위

| 사이트 유형 | 파싱 성공률 (추정) |
|-------------|-------------------|
| 정적 HTML 뉴스 사이트 | 80~90% |
| 한국 주요 뉴스 사이트 (JS 의존) | 40~60% |
| 페이월/보호 사이트 (한경 등) | 5~10% |
| JS 전면 사용 사이트 | ~0% |

실제로 이번 실행에서도 한경(hankyung.com)이 403 에러로 실패한 바 있음.

### 해결 방안 후보

| 방안 | 장점 | 단점 |
|------|------|------|
| **A. newspaper4k로 교체** | newspaper3k의 maintained fork, 다수 버그 수정 | 근본적 JS 미지원 문제는 동일 |
| **B. 본문 품질 필터 추가** | 면책 조항/약관 텍스트 감지하여 걸러냄 | 완벽하지 않음, 엣지 케이스 존재 |
| **C. Gemini에 본문 추출 위임** | URL을 직접 읽지 않고, 검증 단계에서 Grounding이 알아서 검색 | 요약 단계에서 본문 필요 |
| **D. headless 브라우저 (Playwright)** | JS 렌더링 완벽 지원 | GitHub Actions에서 무거움, 실행 시간 증가 |
| **E. 본문 추출 실패 시 해당 기사 제외** | 잘못된 정보 표시 방지 | 기사 수 감소 | //기사중에는 짧은 기사도 있어. '미국과 이스라엘의 공습에 대해 이란이 보복 공습을 시작하다 도널드 트럼프 미국 대통령이 “전례 없는 힘으로 타격하겠다”며 또 다시 이란을 압박했습니다.

1일(현지시간) 트럼프 대통령은 SNS를 통해 “이란이 오늘 매우 강하게, 과거 어느 때보다 더 강하게 공격하겠다고 방금 밝혔다”고 밝혔습니다.

이어 “그들(이란)은 그렇게 하지 않는 것이 좋다. 만약 그렇게 한다면 우리는 전에 본 적 없는 힘으로 그들을 타격할 것”이라며 강경 대응 방침을 재확인했습니다.

로이터통신에 따르면 트럼프 대통령은 플로리다주 팜비치의 마러라고 사저에 머무는 가운데 해당 메시지를 게시한 것으로 전해졌습니다.

앞서 이란 혁명수비대(IRGC)는 이란 최고지도자 아야톨라 알리 하메네이의 사망 소식이 전해진 직후 “최대 규모의 보복 작전을 곧 개시할 것”이라며 미국과 이스라엘에 대한 강력한 대응을 예고했습니다. 이란 유엔 대사 아미르 사이드 이라바니는 “수백 명의 민간인이 공격으로 피해를 입었으며, 보복 공격은 정당한 자위권”이라고 주장했습니다.

이 날 이란 국영방송과 국영 IRNA 통신은 하메네이의 사망 사실을 공식적으로 밝혔습니다. 다만 구체적인 사인에 대해서는 설명하지 않았습니다.
박수유 기자 [aporia@ichannela.com]' 파싱에 실패한 이 기사도 짧은 편이야.  "저작권", "무단 전재", "이용약관" 같은 면책 조항 키워드가 포함되면, 잘못 파싱했다는 걸 판단할 수는 있는데. 제대로 파싱할 수는 없는거야? 정말 어쩔 수 없으면 스킵해야지. 

### 신뢰성 관점

사용자 피드백: "이건 신뢰의 문제야"

**→ 잘못된 본문을 요약하여 표시하는 것은 서비스 신뢰를 심각하게 훼손함.** 파싱 실패 시 엉뚱한 내용을 보여주느니, 차라리 해당 기사를 제외하거나 "본문을 불러올 수 없습니다" 표시가 나음.

## 6. "사실 확인 중" 기사의 출처 링크 표시 문제

### 현재 문제

"사실 확인 중(unconfirmed)" 태그가 붙은 기사 아래에도 grounding_metadata에서 추출된 URL 목록이 표시됨. 이 URL들은 Gemini가 검색은 했지만 확인에 실패한 것들.

사용자 피드백: "충분히 확인하지 못했으면 이 밑 링크는 불필요한 거 아니냐?"

### 분석

맞는 지적. 현재 구조:
- `verified`: 확인 근거 출처 → 링크 표시 **의미 있음**
- `misleading`: 반박 근거 출처 → 링크 표시 **의미 있음**
- `unconfirmed`: AI가 확인 못 함 → 링크 표시 **혼란 유발**

unconfirmed일 때 출처를 보여주면 "이 출처들을 봤는데 확인 못 했다"는 건지, "이 출처들이 관련이 있다"는 건지 사용자가 구분할 수 없음.

### 해결 방안

**A. unconfirmed일 때 출처 숨기기** (권장)
- 템플릿에서 `{% if article.verification_tag != 'unconfirmed' %}` 조건 추가
- 가장 깔끔한 UX

**B. 라벨 변경**
- verified/misleading: "출처 확인하기"
- unconfirmed: "관련 검색 결과" (참고용 성격 명시)

**→ A안 권장: unconfirmed에서는 출처를 아예 숨기는 것이 혼란 방지에 최선.**
